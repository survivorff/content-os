# Scout Report

## 一句话判断

**yes** — 交易所后端+Solana/EVM 双链实战经验的作者极少，"多链不是加适配层"是反常识的实战结论，有硬核内容可写。

## 核心 takeaway

多链扩展最难的不是适配代码，而是三件事：**签名机制差异导致的非原子性风险、链特有的最终性模型如何撕裂你对"交易已确认"的认知、以及你以为通用的"手续费估算"在两条链上根本是两个世界**。

## 目标受众

- 主：Web3 Builder / 中国技术转型者（正在做或考虑做多链产品的后端工程师）
- 次：加密交易者 / 链上打工人（他们关心的是"为什么我的跨链交易延迟/丢单了"，这篇能解释机制本质）

## 独家视角

**强**。市面上讲多链的文章要么是钱包层面的 UX 吐槽，要么是协议层的跨链桥方案。从**交易所后端基础设施**角度拆解多链——尤其是同时做过 Solana 和 EVM 的人——几乎没有。这个视角的稀缺性决定了选题价值。

## 满足的判断条件

- [x] 独家视角
- [x] 硬数据 / 硬证据（真实故障案例即为硬证据）
- [x] 反常识结论（"加个适配层就好"是行业常见错觉）
- [ ] 时效性（非必需，但多链 infra 是持续热点）

## 候选标题

1. **数据型**：《4 年交易所后端，多链扩展的三次真枪实弹——Solana 和 EVM 从来不是一家人》
2. **反差型**：《"不就是加个 adapter 吗？"——做多链交易所四年的三个血泪教训》
3. **身份型**：《一个同时搞过 Solana 和 EVM 基建的人，来拆多链为什么这么难》

推荐：#2，反差+口语化，hook 力度最强，cover 图也好做。

## 风险 / 注意

- **证据具体度**：需要 Frank 提供 3 个坑的真实细节（当时的错误日志、修复方案、损失估计），否则会被认为是泛泛而谈。
- **敏感度**：提到了"交易所"，如果涉及具体公司/项目名，需要确认可否公开。
- **受众地理**：如果主打中文，小红书有必要做；如果主打英文 builder 圈，x-thread 英语版更值。建议先中后英。
