# 论点的解剖学

一个合格的论点长什么样。

---

## 论点 = 论点句 + 支撑 + （可选）过渡

### 论点句

**规则：**
- 一句话
- 最好有一个动作动词（"揭示 / 证明 / 打破 / 绕过"）
- 可证伪（说错了能被反驳）
- 不是疑问，不是感叹

**对比：**

| 坏论点 | 好论点 |
|---|---|
| Web3 不好做 | Web3 后端每 3 个月一次技术范式切换，学习成本是 Web2 的 3 倍 |
| 很多人在 Pump.fun 上亏钱 | Pump.fun 的 Bonding Curve 让前 10% 发币人赚到 80% 的收益 |
| RPC 很重要 | 生产级 Solana dApp 里，RPC 稳定性决定 90% 的用户体验，比合约代码优先级更高 |

坏论点的共同毛病：太泛、不能证伪、没有"啊这点我没想到过"的感觉。

### 支撑

每个论点至少 1 条，最多 3 条。类型：

- **数据**：最硬。带来源。例："Pump.fun 2024 Q4 累计收入 $10.3B（来源：Dune @xxx）"
- **代码 / 截图**：次硬。仅在写技术时用。
- **亲身经历**：软但独家。例："2024 年 3 月我们交易所接 Pump.fun，第一版接口 30% 失败率，花了 2 周优化到 5%"
- **引用他人观察**：最软。只在没其他支撑时用。

支撑的**致命伤**：模糊。

- ❌ "很多案例证明..."
- ❌ "行业里都知道..."
- ❌ "有研究表明..."

都是 smell。找不到具体的数据 / 经历 / 代码，那这个论点就 kill。

### 过渡（可选）

写在当前论点结尾，1 句话，铺垫下一论点。

只在两种情况下需要：
1. 两个论点之间有明确因果关系
2. 读者可能会卡住、需要手把手递过去

其他情况不要写过渡，浪费读者耐心。

---

## 论点之间的关系

3-5 个论点彼此独立但指向同一个 takeaway。

**检查：**
1. 去掉任意一个论点，整体还立得住吗？如果立得住，这个论点可能在凑数。
2. 两个论点的支撑高度重合吗？如果是，合并。
3. 论点的顺序有没有"递进感"？通常应该是：背景 → 核心矛盾 → 解法 → 反思，或者：表象 → 原因 → 深层逻辑 → 行动。

---

## 论点数量

| 数量 | 适用 |
|---|---|
| 3 个 | 轻量话题、小红书向、X Thread 压缩 |
| 4 个 | 标准博客长度 |
| 5 个 | 深度长文 |
| 6+ | 拆系列，或者改清单型 |

超过 5 个别硬写。拆系列。
