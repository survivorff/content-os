# 选题判断标准

## "值得写"的四条线（满足至少 2 条）

### 1. 独家视角（Unique POV）

作者有什么别人没有的？
- 交易所后端经历（meme / 高并发 / 故障）
- Web2 转 Web3 的亲身对比
- 中国开发者看海外生态的"双语视角"
- 在某个具体事件里担任的角色

如果作者只是复述"大家都知道的事"，verdict = kill。

### 2. 硬数据 / 硬证据

有没有可验证的数据或一手材料？
- 链上数据
- 真实交易记录
- 故障时间线
- Benchmark 结果

情绪观点很多，硬数据很少。硬数据是差异化的一半。

### 3. 反常识结论（Counterintuitive）

读者看完会不会"诶我之前想错了"？
- "你以为快 = Solana 赢，其实是三个因素叠加"
- "Web3 赚得多是假象，心理成本你没算"

如果结论就是大家的常识（"Web3 有风险"），verdict = kill。

### 4. 时效 / 钩子（Timeliness）

有没有一个新闻/事件让这个话题"现在写比下个月写强 10 倍"？
- Pump.fun 刚爆出 $10B 收入
- Solana 刚出现大规模 RPC 故障
- 某个 meme 刚造富

时效不是必须，但如果有，是强加分。

---

## Kill 的常见信号

立刻 kill，不要讨价还价：

- 作者自己就 3 分热度，没想清楚
- 话题太泛（"聊聊 Web3"）—— 让作者先收窄
- 作者没有独家视角 + 没有硬数据 + 没有反常识 + 没有时效
- 明显违反人设（比如作者是后端工程师，想写 DeFi 收益攻略——别瞎搞）
- 话题涉及给具体项目带货（Content OS 是作者的 IP 资产，不是广告位）

---

## Maybe 的救活方式

Maybe 通常意味着方向对但角度不好。常见的救活动作：

| 作者原想法 | 救活后的角度 |
|---|---|
| "讲讲 Solana" | "Solana 为什么赢 meme 战争"（收窄） |
| "我在交易所的经历" | "在交易所做 meme 基建的 3 个反常识" |
| "Web3 好难" | "Web2 转 Web3 一年，我多赚了/多亏了多少" |
| "最近看了某书" | "这本书里的 3 个观点，可以用在 Web3 开发" |

救活的本质：**加上作者的独家视角 + 一个清晰的 takeaway**。

---

## 边界 case

- **情绪宣泄**：作者刚被甩/刚离职/刚亏钱。verdict = need-more-info，建议 24 小时后再判。
- **重复选题**：和 `contents/` 里已有文章重合。verdict = maybe，建议换视角或合并成系列。
- **技术太深**：只有 50 个人看得懂。verdict = yes，但平台只选 blog。
- **过度八卦**：全靠未证实信息吃瓜。verdict = kill，保护作者的信用。
