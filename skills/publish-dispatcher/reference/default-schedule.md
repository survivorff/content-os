# 默认发布排期

inbox 的 frontmatter 没写 `schedule` 时用这些默认规则。

---

## 默认排期逻辑

**D-Day**（文章确定要发的那天）按以下规则排：

| 平台 | 时间 | 理由 |
|---|---|---|
| blog | D-Day 10:00（UTC+8） | 工作日早上推 RSS / 搜索 |
| x-thread | D-Day 20:00（UTC+8） | 晚上黄金时段（中国 + 欧洲） |
| xiaohongshu | D+1 12:00（UTC+8） | 第二天午休时段，不和 X 打架 |
| douyin | D+1 20:00（UTC+8） | 第二天晚间高活跃 |

错峰原则：**博客先发，社交媒体分时段二次扩散**。

---

## 调度策略

### 工作日 vs 周末

- **默认避开周末**：博客 + X 的阅读量周末低 50%+
- **周五发 → 博客和 X 放到下周二**

D-Day 如果是周五、周六、周日：
```
blog:        下周二 10:00
x-thread:    下周二 20:00
xiaohongshu: 下周三 12:00
douyin:      下周三 20:00
```

### 同天多篇

一天最多：
- 1 篇博客
- 1 条 X Thread
- 1 篇小红书
- 1 个抖音

如果 queue 里已有今天同平台的内容 → 顺延到下一个空位。

### 时效性内容

如果 scout.md 里 `time_sensitive: true`（例如刚爆出的新闻），忽略默认排期，立刻发。

---

## 用户覆盖

用户在 inbox 里写的 `schedule` 永远优先。

```yaml
schedule:
  blog: 2026-05-10T10:00:00+08:00
  x: 2026-05-10T20:00:00+08:00
  xiaohongshu: 2026-05-11T12:00:00+08:00
```

用户只写部分 → 没写的按默认。

---

## 时区处理

- **默认时区**：Asia/Shanghai (UTC+8)
- **存储时区**：全部 ISO 8601 with offset
- 用户写 `2026-05-10T10:00:00` 无时区 → 按 UTC+8 解释

---

## X Thread 发多少才合理

X 上最佳节奏：
- **1-2 个 Thread / 周**（质量为主）
- 同一天不要发 2 个 Thread
- 两个 Thread 至少隔 48 小时

queue 里已有的如果冲突，顺延到下个空位。

---

## 小红书 / 抖音

没有硬限流规则，但建议：
- 小红书 1-3 篇/周
- 抖音按录制能力，不强求

---

## 节假日

`reference/holidays.md`（如果有）列出特殊日期。节假日不发技术内容，阅读量低 40%+。

目前的节假日列表（简化）：
- 春节（农历正月 1-7）
- 国庆（公历 10.1-10.7）
- 5.1
- 中秋

这些日期排期要手动确认，不要自动化。
