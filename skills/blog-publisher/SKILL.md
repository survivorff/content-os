# Blog Publisher

你是唯一有"真正发布能力"的 skill。其他平台（X / 小红书 / 抖音）是生成 + 提醒，你这里是**把文件真的放到 blog 仓库 + git push 触发部署**。

---

## 协议

### 1. 前置检查

- `contents/<slug>/meta.json` 存在，且 `quality_gate.blog.status` ∈ {approved, approved_with_notes}
- `generated/<slug>/blog.md` 存在
- `meta.json.published.blog` 应该是 `null` 或 `pending`。如果已经是 ISO 时间戳，说明**已发过**，**不要重发**。

### 2. 环境探测

blog 仓库在 VM 上的固定位置：`/home/admin/blog`

```bash
# 验证
test -d /home/admin/blog/src/data/blog
```

如果不在那里：停下来，告诉用户去按 `docs/setup-openclaw.md` 克隆 blog 仓库，不要自己瞎 clone。

### 3. 复制文件

```bash
SRC=/home/admin/content-os/generated/<slug>/blog.md
DST=/home/admin/blog/src/data/blog/<slug>.md
cp "$SRC" "$DST"
```

**规则**：
- slug 必须和 generated 文件所在目录名一致
- 如果 `$DST` 已经存在（之前发过），先 diff 一眼差异再覆盖。差异太大（>50% 行变化）停下来问用户是不是 slug 冲突了。

### 4. 校验 frontmatter

打开 `$DST`，确认以下字段都有且合理：

- `author` = "Frank Fu"
- `pubDatetime` 有值，ISO 8601
- `title` 有值
- `slug` 和文件名一致
- `tags` 至少一个
- `draft: false`（**关键**，如果是 true 就不会上线）
- `description` 有值

任何一项缺失或错误 → 停下来报错，不要硬发。

### 5. Git commit + push

```bash
cd /home/admin/blog
git pull --rebase origin main 2>&1  # 先拉，避免冲突
git add src/data/blog/<slug>.md
git commit -m "content: <slug>

Generated by content-os agent on $(date -Iseconds)
Source: github.com/survivorff/content-os/commit/<current-hash>"
git push origin main
```

获取 content-os 当前 commit hash 的方法：

```bash
cd /home/admin/content-os
CONTENT_OS_HASH=$(git rev-parse --short HEAD)
```

### 6. 等部署（可选但推荐）

push 完后，blog 的 GitHub Actions 部署大概需要 1-2 分钟。你可以：

- A) 不等，直接汇报"已触发部署，约 2 分钟后生效"
- B) 每 30s 拉 `https://blog.frankfu.cloud/posts/<slug>` 看 HTTP 状态，变 200 再报"已上线"

**推荐 A**，B 会占用 agent 时间。

### 7. 更新 meta.json

```json
{
  "published": {
    "blog": "2026-05-10T10:01:15+08:00"
  },
  "blog_url": "https://blog.frankfu.cloud/posts/<slug>"
}
```

也更新 `queue/schedule.json` 里对应条目的 `status` 为 `executed`，`executed_at` = 当前时间。

### 8. 输出

向 orchestrator 汇报：

```
✅ Blog published:
- URL: https://blog.frankfu.cloud/posts/<slug>
- Commit: https://github.com/survivorff/blog/commit/<hash>
- Deploy: triggered (ETA 2 min)
```

---

## 铁律

1. **绝不在 quality-gate rejected 时发布**
2. **绝不在 meta.json.published.blog 已有时间戳时重发**
3. **绝不修改 blog 仓库里除 src/data/blog/<slug>.md 之外的文件**
4. **绝不改 draft: true 为 false**（作者明确设 draft 就是不想发）
5. **绝不用 git push --force**
6. **git push 失败时立刻停下来并报错给用户**，不要 retry 更不要自己解冲突

---

## 常见失败

### push 被 rejected（远程有新 commit）

先 `git pull --rebase`，再 push。失败 → 告诉用户远程仓库有冲突，让他手动处理。

### blog 部署失败

不是你的问题，是 blog 仓库 Actions 失败。告诉用户去看 blog repo 的 Actions 日志。你的责任到 push 为止。

### slug 冲突

已经有同名 slug → 两种可能：
1. 这是同一篇文章的更新（正常，继续覆盖）
2. 这是两篇不同文章撞名（严重，让用户改 slug）

让用户判断。
