# Blog Publisher

你是唯一有"真正发布能力"的 skill。其他平台（X / 小红书 / 抖音）是生成 + 提醒，你这里是**把文件真的放到 blog 仓库 + git push 触发部署**。

---

## 协议

### 1. 前置检查

- `contents/<slug>/meta.json` 存在，且 `quality_gate.blog.status` ∈ {approved, approved_with_notes}
- `generated/<slug>/blog.md` 存在
- `meta.json.published.blog` 应该是 `null` 或 `pending`。如果已经是 ISO 时间戳，说明**已发过**，**不要重发**。

### 2. 环境探测

blog 仓库在 VM 上的固定位置：`/home/admin/blog`

```bash
# 验证
test -d /home/admin/blog/src/data/blog
```

如果不在那里：停下来，告诉用户去按 `docs/setup-openclaw.md` 克隆 blog 仓库，不要自己瞎 clone。

### 3. 复制文件

```bash
SRC=/home/admin/content-os/generated/<slug>/blog.md
DST=/home/admin/blog/src/data/blog/<slug>.md
cp "$SRC" "$DST"
```

**规则**：
- slug 必须和 generated 文件所在目录名一致
- 如果 `$DST` 已经存在（之前发过），先 diff 一眼差异再覆盖。差异太大（>50% 行变化）停下来问用户是不是 slug 冲突了。

### 4. 校验 frontmatter

打开 `$DST`，确认以下字段都有且合理：

- `author` = "Frank Fu"
- `pubDatetime` 有值，ISO 8601
- `title` 有值
- `slug` 和文件名一致
- `tags` 至少一个
- `draft: false`（**关键**，如果是 true 就不会上线）
- `description` 有值

任何一项缺失或错误 → 停下来报错，不要硬发。

### 4b. Git commit（push 之前先 commit）

```bash
cd /home/admin/content-os
CONTENT_OS_HASH=$(git rev-parse --short HEAD)

cd /home/admin/blog
git pull --rebase origin main 2>&1  # 先拉，避免 non-fast-forward
git add src/data/blog/<slug>.md
git commit -m "content: <slug>

Generated by content-os agent
Source: github.com/survivorff/content-os/commit/$CONTENT_OS_HASH"
```

### 5. Pre-push 验证（新增，防止 push 到一半挂掉）

在 commit 后、真 push 前，**先跑 dry-run**：

```bash
cd /home/admin/blog
git push --dry-run origin main 2>&1
```

解析输出：

- 看到 `remote: Permission to ... denied` 或 `403` → **PAT 权限不足**
  - 诊断 message（必须一字不差告诉用户）：
    ```
    ⚠️ Blog push 权限被拒 (403)
    原因：fine-grained PAT 对 blog 仓库的 Contents 可能不是 Read and write
    修复步骤：
      1. 打开 https://github.com/settings/tokens?type=beta
      2. 找到当前 PAT（名字有 content-os 或类似）
      3. Repository access → 确认勾选 blog
      4. Repository permissions → Contents: 改成 Read and write
      5. 保存
      6. 告诉我"PAT 改好了"，我自动 retry
    ```
  - **不要重试 push**。直接进入 step 8（记录失败状态）。
- 看到 `401 Unauthorized` → **PAT 已过期**
  - 让用户重新生成 PAT，update `.env`，然后 retry
- 看到 `rejected (non-fast-forward)` → **远程有新 commit**
  - 执行 `git pull --rebase origin main`，成功则 retry dry-run，失败则报告冲突
- 干净 → 继续 step 6

### 6. 真正 push

```bash
cd /home/admin/blog
git push origin main 2>&1
```

如果这次失败（dry-run 通过了但真 push 失败，罕见）→ 按 step 5 的同样方式分类记录。

### 7. 等部署（可选但推荐）

push 完后，blog 的 GitHub Actions 部署大概需要 1-2 分钟。你可以：

- A) 不等，直接汇报"已触发部署，约 2 分钟后生效"
- B) 每 30s 拉 `https://blog.frankfu.cloud/posts/<slug>` 看 HTTP 状态，变 200 再报"已上线"

**推荐 A**，B 会占用 agent 时间。

### 7. 更新 meta.json（成功时）

```json
{
  "published": {
    "blog": "2026-05-10T10:01:15+08:00",
    "blog_url": "https://blog.frankfu.cloud/posts/<slug>/",
    "blog_commit": "github.com/survivorff/blog/commit/<hash>"
  }
}
```

### 7b. 更新 meta.json（失败时，一定要写）

如果 push 失败，**仍然写 meta.json**，用这个模板：

```json
{
  "published": {
    "blog": "failed",
    "blog_failure_reason": "git push 403 - PAT likely missing Contents: Read and write for blog repo",
    "blog_failure_code": "permission-denied | pat-expired | rebase-conflict | network",
    "blog_retry_command": "cd /home/admin/blog && git push origin main",
    "blog_local_commit": "<已经 commit 的 hash，方便 retry 时找>"
  }
}
```

也更新 `queue/schedule.json` 里对应条目：`status: "failed"` + `error: "<具体原因>"`。

这样下次 agent 或用户 retry 时，能直接从失败点 pick up，不用重新跑整条 pipeline。

### 8. 输出

向 orchestrator 汇报：

```
✅ Blog published:
- URL: https://blog.frankfu.cloud/posts/<slug>
- Commit: https://github.com/survivorff/blog/commit/<hash>
- Deploy: triggered (ETA 2 min)
```

## 铁律

1. **绝不在 quality-gate rejected 时发布**
2. **绝不在 meta.json.published.blog 已有时间戳时重发**
3. **绝不修改 blog 仓库里除 src/data/blog/<slug>.md 之外的文件**
4. **绝不改 draft: true 为 false**（作者明确设 draft 就是不想发）
5. **绝不用 git push --force**
6. **git push 失败时立刻停下来并报错给用户**，不要 retry 更不要自己解冲突

---

## 常见失败

### push 被 rejected（远程有新 commit）

先 `git pull --rebase`，再 push。失败 → 告诉用户远程仓库有冲突，让他手动处理。

### blog 部署失败

不是你的问题，是 blog 仓库 Actions 失败。告诉用户去看 blog repo 的 Actions 日志。你的责任到 push 为止。

### slug 冲突

已经有同名 slug → 两种可能：
1. 这是同一篇文章的更新（正常，继续覆盖）
2. 这是两篇不同文章撞名（严重，让用户改 slug）

让用户判断。
