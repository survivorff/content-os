# 小红书正文结构

---

## 标准结构（500-800 字）

```
[开场 1-2 段]
身份 + 吸引注意力

[主体 3-5 个小标题]
每个小标题用 emoji 开头
每个段落 1-3 句话

[结尾 1 段]
互动引导 + CTA
```

---

## 开场模板

### 模板 1 — 身份锚定

```
💼 [我的具体身份 + 时长]。

[最近发生了一件 X 的事]。

今天把 [要讲的东西] 整理一下，[给谁] 参考 ⬇️
```

**示例：**
> 💼 在交易所做 Web3 后端 2 年。
>
> 最近有朋友问我转 Web3 该注意什么。
>
> 今天把真实踩过的 5 个坑写下来，想转的兄弟参考 ⬇️

### 模板 2 — 反差开场

```
🤔 很多人以为 [A]。
实际上 [B]。

作为 [身份]，今天讲讲真相 ⬇️
```

### 模板 3 — 数字冲击

```
💰 [一个震撼数字 + 事件]。

[你不敢相信？]

我来拆解一下 ⬇️
```

## 主体结构

每个小标题一个章节。小标题格式：

```
[emoji] [小标题名]（5-10 字）

[1-3 句话的论点 / 观察]

[1-2 句话的支撑]
```

**常用 emoji：**
- 💰 钱 / 收入
- 🧠 思考 / 决策
- ⚠️ 坑 / 警告
- ✅ 正确做法 / 建议
- 🔥 重点 / 爆点
- 📊 数据 / 对比
- 💡 洞察 / tips
- 🚀 行动 / 突破

**示例小标题：**
- 💰 真相 1：钱多，但赚得心惊胆战
- 🧠 真相 2：什么时候该换技术栈
- ⚠️ 坑点 1：我踩过最恶心的 RPC 问题
- 📊 对比：Web2 和 Web3 后端的 5 个不同

---

## 段落节奏

**短、短、短。**

```
错误示例（太长）：
在交易所工作的这两年里，我逐渐意识到 Web3 后端和 Web2 后端在很多方面都有本质的区别，这种区别不仅仅体现在技术栈上，更多的是体现在思维方式和业务流程上。

正确示例：
💡 Web3 和 Web2 后端，差的不是技术栈。

是思维方式。

举个例子：Web2 我改一个 bug，用户最多不开心；Web3 我改一个 bug，可能直接损失几十万美金。
```

每段 **1-3 句话**，超过 4 句话就拆。

## 禁止

- ❌ 段落超 4 句
- ❌ 没有小标题的大段文字
- ❌ 贴大段代码
- ❌ 引用英文论文或文档
- ❌ 用"然而"、"因此"、"综上所述"等书面词
- ❌ 没有 emoji 的小标题

---

## 总字数

目标 500-800 字。

- < 500：内容太薄，读者觉得"不值点开"
- > 800：小红书用户注意力撑不住，后半段没人看
