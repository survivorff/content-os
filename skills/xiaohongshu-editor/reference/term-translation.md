# 术语翻译表

小红书用户不是技术人。专业术语要"翻译"成生活化语言。

**原则：宁可不精确，不能让读者懂不了。**

---

## 通用翻译对照

| 技术术语 | 场景 | 小红书说法 |
|---|---|---|
| RPC | 通用 | 区块链节点服务（一种网络请求） |
| MEV | 通用 | 有人在你交易前后下单赚差价 |
| Bonding Curve | 发币 / 定价 | 一个自动算价格的公式 |
| Compute Unit | Solana 交易费 | 像 gas 费一样要算清楚的资源 |
| Slippage | 交易 | 下单价和成交价之间的差 |
| Gas | 通用 | 链上手续费 |
| Bridge | 跨链 | 跨链转资产 |
| Liquidity Pool | AMM / DEX | 自动交易的资金池 |
| Smart Contract | 通用 | 部署在链上的程序 |
| Validator | 共识 / PoS | 区块链上投票打包交易的节点 |
| Consensus | 通用 | 节点之间怎么达成一致 |
| On-chain / Off-chain | 通用 | 链上 / 链下（仍保留，但加解释） |
| Fork | **软件版本** | 分叉（一条链被复制成两条，用于升级或历史事件） |
| Fork | **RPC 节点场景** | 暂时没同步最新区块的节点（会返回旧数据） |
| Token | 通用 | 代币 |
| Mint | 通用 | 铸造（创建新代币） |
| Burn | 通用 | 销毁代币 |
| Staking | 质押 | 质押（把币锁住换收益） |
| Airdrop | 通用 | 空投（免费发币给用户） |
| P50 / P99 / P999 | 性能指标 | 50%/99%/99.9% 情况下的速度（排序后的分位数） |
| Throughput / TPS | 性能指标 | 吞吐量（每秒能处理多少笔） |
| Downtime | 运维 | 宕机时间 / 挂掉的时间 |
| Indexer | 数据索引 | 把链上数据整理成数据库（便于查询） |

**重要**：同一个术语在不同 context 可能要不同译法。写的时候先判断 context 再查表。

## 技术名词保留英文的条件

以下情况保留英文：

1. **中文翻译反而让人困惑**（RPC、MEV 这种，中文译名 10 个人看懂 3 个）
2. **业内已有约定俗成**（Web3、Solana、Ethereum 这种）
3. **专有名词 / 品牌名**（Pump.fun、MetaMask）

保留英文时**首次出现给括号解释**：

> RPC（区块链节点服务，dApp 靠它和链通信）

## 翻译的底线

翻译可以让小红书读者读懂，但**不能误导**。

**错误翻译示例：**

❌ 把 MEV 翻译成"抢跑" —— MEV ≠ 抢跑，抢跑只是 MEV 的一种
❌ 把 RPC 翻译成"链上 API" —— 技术上不准确

**正确做法：**

✅ "MEV（交易排序里的套利机会，抢跑是其中一种）"
✅ "RPC（和区块链通信的服务接口）"

---

## 验证：翻译够不够到位

写完后用这个问法自查：

**"一个完全不懂 Web3 的互联网产品经理，能看懂这段吗？"**

看不懂 → 继续降维。

**"一个 Web3 资深开发者，看到这段会觉得我在瞎说吗？"**

会 → 翻译牺牲了准确性，需要加括号补充。

两个检查都过，说明降维到位。
